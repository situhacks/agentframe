# Post 1 — FINAL: The Agent Boundary Problem

Assembled from the post's locked ingredients per the campaign manifest (`body-copy`, `image-prompts`).

## Body Copy (locked from body-copy-v3.md)

I do not want an AI agent that can do everything.

I want one that knows exactly when to leave me alone and when to speak up.

The demos always show the fun version: agent opens a browser, clicks through five apps, updates a record, sends a message, maybe makes coffee by the end. But once it touches real tools, the question changes.

It is not "can the model do the task?" It is: what can it read, what can it write, when does it ask, and who owns the mistake?

A scheduled prompt is useful, but it is not really an agent. The other extreme is giving a vague agent broad access and hoping the guardrails hold. That feels backwards to me.

The useful middle is less flashy: a clear job, a small tool surface, an escalation path, and an audit trail.

In the enterprise, that is governance. In the IDE, it is the moment you stop adding tools and start deciding what the agent should never be allowed to do.

My POV: the next serious agent capability is not more autonomy. It is restraint.

Where would you draw the line between useful automation and too much autonomy?
