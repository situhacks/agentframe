# Operator context — canonical shapes

The canonical shape of each operator-context surface: positioning, profile, design-language, and voice. These are the single source the setup process generates from — not files to copy by hand. First-run generation runs [`operator-context-setup.md`](../../process/operator-context-setup.md), which writes the operator's filled-in files to `library/context/operator/` (gitignored — operator content stays private).

| Schema | What it is | How it's filled |
|---|---|---|
| `positioning.md` | Strategic stance / content pillars | Setup interviews the operator against this shape. |
| `profile.md` | Who the operator is (operator-self) | Setup interviews the operator against this shape. |
| `design-language.md` | How rendered deliverables look | Setup interviews the operator against this shape. |
| `voice/` | Writing voice (a folder, not one file) | **Built by process, not form-filling — run [`voice-setup.md`](../../process/voice-setup.md).** It builds voice from real writing (samples → mine corpus → taste interview → compile), because voice comes from examples, not self-description. |

Voice needs a process, not form-filling — that's what `voice-setup.md` is for.
